**Pattern 1 — function/filter definitions** (verified AST, see 02-RESEARCH.md "Pattern 1"): both
`function Name { }` and `filter Name { }` parse to `function_statement`. The name is a POSITIONAL
child of kind `function_name` — NOT a named field (shell.rs's `field_text(&child, "name", ...)`
does NOT port; use `child_text(&child, "function_name", ctx.bytes)` instead — this is the #1
anti-pattern flagged in research). Walk `root`'s descendants for `function_statement` nodes, emit a
`SymbolKind::Function` symbol per match via `make_symbol` with `Visibility::Unknown` (D-03) and
descriptors `namespaces + [Descriptor::Method { name, disambiguator: String::new() }]`.

**Pattern 2 — class + method + inheritance** (verified AST, "Pattern 2"): `class_statement`'s first
`simple_name` child (no field) is the class name; emit `SymbolKind::Class`. If a SECOND `simple_name`
child exists (only present when a `:` token is present), it is the base class — emit
`RefRole::IsImplementation` via `push_ref` (Claude's discretion, confirmed unambiguous per research).
Each `class_method_definition` child's own first `simple_name` child is the method name — emit
`SymbolKind::Method` with descriptors `namespaces + [Descriptor::Type(class_name), Descriptor::Method{name, disambiguator: String::new()}]`.
**Gotcha (do not violate):** do NOT write one combined tree-sitter query with an optional
`(simple_name)? @base_name` capture — verified to produce spurious duplicate matches (3 matches for
2 classes). Walk `class_statement`'s direct named children manually instead, exactly like
`shell.rs`'s manual `collect_symbols` walk.

**Pattern 3 — calls, both forms** (verified working queries, "Pattern 3"): run
`collect_call_references` TWICE and concatenate the results —
```
const CMDLET_CALL_QUERY: &str = r#"(command command_name: (command_name) @callee)"#;
const MEMBER_CALL_QUERY: &str = r#"(invokation_expression [(variable) (type_literal)] @qualifier (member_name (simple_name) @callee))"#;
```
`collect_call_references` already supports the optional `@qualifier` capture — zero changes needed to
`support.rs`. Note: `CMDLET_CALL_QUERY`'s pattern requires the `command_name` FIELD's node to
literally be kind `command_name` (bare) — dot-sourcing's wrapped `command_name_expr(command_name)`
shape does NOT match this pattern, so dot-sourced commands are automatically excluded from this query
already.

**Pattern 4 — import classification, ALL THREE forms are `command` nodes** (verified, "Pattern 4"):
the grammar has no dedicated import node; classify by `command_name` TEXT using two additional
queries, matched BEFORE emitting the generic calls list (see double-count guard below):
```
const IMPORT_ARG_QUERY: &str = r#"(command command_name: (command_name) @cmd command_elements: (command_elements (generic_token) @arg))"#;
const DOT_SOURCE_QUERY: &str = r#"(command (command_invokation_operator) command_name: (command_name_expr (command_name) @path))"#;
```
Run `IMPORT_ARG_QUERY` with a raw `Query`/`QueryCursor` (not `collect_call_references` — you need the
`@cmd`+`@arg` pair, not just `@callee`): for each match, if `@cmd` text (case-insensitive) equals
`"Import-Module"`, emit `RefRole::Import` with name = the FIRST `@arg` in that match's
`command_elements`. If `@cmd` text (case-insensitive) equals `"using"` AND the first `@arg` in that
match equals `"module"`, emit `RefRole::Import` with name = the SECOND `@arg`. Run `DOT_SOURCE_QUERY`
separately: each `@path` capture is a dot-sourced file path — emit `RefRole::Import` with that text
as name.
**Double-count guard (Pitfall 2 — mandatory):** collect the byte offset (`start_byte()`) of every
`command_name` node classified as an import above into a `HashSet<usize>`. After running
`CMDLET_CALL_QUERY` via `collect_call_references`, filter OUT any resulting `Call` reference whose
`occ.byte` is in that set (both queries capture the exact same `command_name` node position for
`Import-Module`/`using`, so byte-offset equality is exact — no fuzzy matching needed).
