<objective>
Ship the Erlang extractor end-to-end. The defining technical fact (verified): every
`Name(Args) -> Body.` clause — even semicolon-chained multi-clause pattern matches of the SAME
name+arity — is its OWN top-level `fun_decl` node; the grammar never pre-groups them. D-06's
"one function symbol per name/arity" is closed via the SAME dedup-by-SCIP-id mechanism `objc.rs`
already uses for `@interface`/`@implementation` — NOT a separate grouping pass — now made correct by
a MANDATORY (not discretionary, unlike Elixir) `Descriptor::Method` arity disambiguator.

Purpose: Third of the four LANG-12 BEAM-family languages. `-export` is the cleanest visibility signal
of the whole phase (arity-qualified in the grammar itself, like Go's capitalization convention).
Output: `src/extract/erlang.rs`, updated `src/lang.rs`/`Cargo.toml`/both bindings `Cargo.toml`s,
`eval/corpus/erlang/scoped_call/`, updated docs row.
</objective>

**Symbol/reference collection** (verified AST, 04-RESEARCH.md "Erlang" Architecture Patterns +
Code Examples sections):
- First pass over `source_file`'s children: collect every `export_attribute funs: (fa fun: (atom) arity: (arity value: (integer)))` pair into a `HashSet<(String, String)>` of `(name, arity_string)` — this IS structurally arity-qualified in the grammar (unlike Elixir, no counting needed; read the `arity value:` field directly).
- Second pass: for each top-level `fun_decl clause: (function_clause name: (atom) args: (expr_args args: (...)) body: (clause_body ...))`, read the clause's `name:` atom text and compute arity as the COUNT of `args:` field children within `expr_args` (no explicit arity field on `function_clause` itself — must count). Visibility = `Public` if `(name, arity.to_string())` is in the export set, else `Private`. Emit `SymbolKind::Function` + `Descriptor::Method { name, disambiguator: arity.to_string() }` via the standard `seen: HashSet<String>` + `push_symbol` dedup wrapper — clauses of the same name+arity collapse to the FIRST occurrence automatically; different arities keep distinct SCIP ids and both survive.
- `-import(Module, [...]).` / `-include("path").` / `-include_lib("path").` → each emits one
  `RefRole::Import` (for `-import`, one Import per listed function name with `from_path` = the module
  atom; for `-include`/`-include_lib`, one Import with the literal path string as both name and
  `from_path`).
- Calls:

  ```rust
  ]
  "#;
  ```
  Run via `collect_call_references` — note the field name is `expr:`, NOT `function:` (a verified
  divergence from every C-family/Lua-family CALL_QUERY in this codebase; do not assume `function:`).
