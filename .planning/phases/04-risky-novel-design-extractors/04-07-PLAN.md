<objective>
Ship the Julia extractor end-to-end — the first of the two novel-dispatch scripting languages (D-09:
Julia before R). D-01's ceiling is the phase's central discipline: multiple dispatch means Call refs
stay `NameOnly` fan-out, never method-signature-matched — this is a resolver-tier concern explicitly
out of milestone scope. The one non-obvious grammar fact this plan MUST get right: `f(x) = ...`
(short-form function definition) is an `assignment` node, NOT a `function_definition` — missing this
arm silently drops every short-form Julia function.

Purpose: Ship LANG-02 with the multiple-dispatch ceiling honestly documented (never guessed past), and
apply the arity-disambiguator Descriptor pattern (proven in 04-04/04-05) to close the *majority*
same-name/different-arity collision case, while documenting the residual same-arity/different-type gap.
Output: `src/extract/julia.rs`, updated `src/lang.rs`/`Cargo.toml`/both bindings `Cargo.toml`s,
`eval/corpus/julia/scoped_call/`, updated docs row.
</objective>

**Namespace derivation**: `julia_namespaces(file)` — prefer the FIRST top-level `module_definition`'s
name; fall back to path-derived namespace (strip `.jl`, strip `src/`, split on `/`) when absent (same
"prefer explicit declaration, fall back to path" shape as `fortran_namespaces`).

**Symbol collection** (verified AST, 04-RESEARCH.md "Julia" Code Examples section — dispatch on each
top-level child's node kind):
- `module_definition name: (identifier)` → `SymbolKind::Module` + `Descriptor::Namespace`; recurse
  into children under the extended namespace.
- `using_statement (selected_import (identifier) (identifier))` / `import_statement (selected_import ...)`
  → for each `selected_import`, the FIRST identifier is the module, the REST are selected names; emit
  one `RefRole::Import` per selected name (`from_path` = the module name).
- `struct_definition (type_head (identifier)) (identifier) (identifier)` / `abstract_definition (type_head (identifier))`
  → `SymbolKind::Struct` (both — no separate abstract-type kind, document this mapping in the module
  doc rather than silently conflating).
- `function_definition (signature (call_expression (identifier) (argument_list ...))) (return_statement ...)`
  → LONG-form `SymbolKind::Function`. Compute arity as the count of children in the inner
  `call_expression`'s `argument_list`. Emit `Descriptor::Method { name, disambiguator: arity.to_string() }`.
- `assignment (call_expression (identifier) (argument_list (identifier))) (operator) (binary_expression ...)`
  → SHORT-form `SymbolKind::Function` — SAME kind, SAME disambiguator scheme as the long form, just a
  DIFFERENT top-level node kind (`assignment` where the LHS child is a `call_expression`, not any
  bare-value assignment). Do not miss this arm.
- `macro_definition (signature (call_expression (identifier) (argument_list (identifier)))) (identifier)`
  → `SymbolKind::Function` + `Descriptor::Macro(name)` (reuses the C `#define` identity precedent
  directly, per Don't Hand-Roll).

Use the standard `seen: HashSet<String>` + `push_symbol` dedup wrapper for every emitted symbol (this
IS load-bearing for the arity-disambiguated Function symbols, same as Elixir/Erlang).

**Calls**:
```rust
const CALL_QUERY: &str = r#"
[
  (call_expression (identifier) @callee (argument_list))
  (call_expression (field_expression value: (identifier) @qualifier (identifier) @callee) (argument_list))
]
"#;
```
Note: `call_expression`'s callee/field_expression children are POSITIONAL (no `function:` field, per
research — do not add a field prefix that doesn't exist in this grammar).

**Macro-call references** (separate manual walk, NOT part of `CALL_QUERY` — `macro_identifier` is a
distinct node kind from plain `identifier`): recursively walk for `macrocall_expression (macro_identifier (identifier)) (argument_list ...)`
nodes; emit a `RefRole::Call` reference using the inner `identifier`'s text (already `@`-free per the
grammar's own tokenization — verify against the AST dump, do not manually strip a literal `@` unless
the captured text actually contains one).

**Wiring**: `src/extract/mod.rs` — `#[cfg(feature = "julia")] pub mod julia;` /
`#[cfg(feature = "julia")] pub use julia::JuliaExtractor;` (alphabetical, near `java`/`kotlin`). Update
`src/extract/dispatch.rs` with the matching `use` + `Language::Julia => JuliaExtractor.extract(source, file),`
arm.

`definition_bindings(&defs)`.
